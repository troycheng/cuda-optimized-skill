# 面向使用者的 README 重写设计

## 目标

重写中英文 README，使读者无需理解项目内部状态机或 CLI，就能回答以下问题：

1. 这是什么项目；
2. 可以解决哪些 CUDA 与 GPU 性能问题；
3. 用户需要提供什么；
4. AI 会执行哪些工作；
5. 项目如何判断优化是否有效；
6. 最终会交付什么；
7. 哪些操作不会自动执行；
8. 项目经过了哪些实际测试。

README 面向使用者和项目评估者。AI 执行协议、CLI、JSON schema、发布命令与完整
验收记录不进入主阅读路径，由专门文档承载。

## 写作原则

- 正文使用中立、克制的项目说明语气。
- 自然语言任务描述只出现在「使用示例」中。
- 标题使用普通语言；必要的专业术语在首次出现时解释作用。
- 保留 CUDA、CUTLASS、Triton、kernel、workload、profiling、成对 A/B 测试和
  置信区间等领域术语，不使用「晋级」「终局状态」「旁路证据」「可信边界」等内部词汇。
- 先解释能力和结果，再介绍实现方法与证据。
- 中英文 README 信息对等，分别按母语习惯写作，不逐句直译。

## 信息架构

两份 README 使用相同的章节顺序：

1. **项目是什么**
   - 说明这是面向 Codex 的 CUDA 性能优化项目，核心能力以 skill 提供。
   - 覆盖单个 kernel 和完整 GPU workload。
2. **可以解决哪些问题**
   - 用表格列出 kernel 优化、完整 workload 优化、真实 workload 验证和已有 NCU
     report 分析。
   - 每行说明适用场景、AI 执行内容和主要结果。
3. **需要提供什么**
   - 可运行代码或 workload、测试环境、正确性基准、性能目标与业务约束。
   - 说明真实 workload 必须由用户提供，AI 不下载或编造。
4. **AI 会如何执行**
   - 使用一张高层 Mermaid 图：理解目标、检查环境、建立 baseline、profiling、修改、
     成对评测、保留修改或恢复原实现。
   - 不展示内部命令、checkpoint 或文件注册步骤。
5. **如何确认优化结果**
   - 解释正确性、成对 A/B 测试、置信区间、性能门槛和约束检查。
   - 说明证据不足、结果变差或约束失败时保留原实现。
6. **最终会得到什么**
   - 修改后的代码、瓶颈分析、性能数据、正确性与约束结果、未采用方案及原因、宿主机建议。
7. **修改范围与安全限制**
   - 项目内代码和用户提供的隔离环境可在声明范围内修改。
   - 驱动、权限、频率、功耗和宿主机系统配置只给建议，不自动执行。
   - 第三方 reviewer 只提供意见，不能决定是否保留修改。
8. **使用示例**
   - 提供 3 至 4 个用户对 Codex 的自然语言任务示例。
   - 覆盖 kernel、完整 workload、NCU report 和计算预算。
9. **测试情况与兼容性**
   - 用简表保留 RTX 5090、完整 workload、NCU report 和 CPU 测试的可理解结论。
   - 镜像 SHA、逐项 hash、完整命令和详细验收记录通过链接提供。
10. **安装与进一步文档**
    - 说明 skill 的安装入口与所需运行环境。
    - 链接 AI 执行协议、完整 workload 示例、兼容性说明、优化方法目录、测试说明和许可证。

## 内容迁移

| 当前 README 内容 | 处理方式 |
|---|---|
| `setup`、`open-iter`、`close-iter`、`resume`、`finalize` 命令 | 从 README 删除，链接到 `SKILL.md` 和 walkthrough |
| Workload controller CLI 与 JSON 细节 | 从 README 删除，链接到完整示例 |
| 安装器和双远端发布命令 | README 只保留安装入口；发布流程移出使用者路径 |
| 三张内部流程图 | 合并为一张面向使用者的高层流程图 |
| 预算完整参数表 | 改成 `quick`、`balanced`、`thorough` 的用途和大致时间表 |
| 状态名和 artifact 目录树 | 改成最终交付物说明；详细格式链接到 AI 执行协议 |
| 镜像 SHA、逐项 hash 与完整验收数字 | README 保留关键结论，详细证据链接到测试说明 |
| 仓库发布与 upstream 保护 | 移到维护者文档，不放在使用者 README 正文 |

## 验证标准

- 中英文 README 使用相同章节顺序和能力集合。
- 主阅读路径不包含 orchestrator、workload controller、strategy memory 或发布工具的
  手工执行命令。
- README 最多保留一张 Mermaid 图；自然语言任务描述只出现在「使用示例」。
- 开篇在三段内说明项目定义、用途和能力程度，不出现版本演进记录。
- 关键技术边界与现有实现一致，不夸大自动化范围。
- 现有测试改为验证新的使用者信息架构，同时继续校验预算、能力、测试结果和双语一致性。
- 完整测试、skill validator、脚本语法检查和 README 链接检查全部通过。
